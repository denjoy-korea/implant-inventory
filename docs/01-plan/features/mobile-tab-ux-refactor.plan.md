# 모바일 탭 UX 재설계: 재고-발주 역할 분리 의사결정

> **Summary**: 모바일 환경에서 "재고" 탭과 "주문" 탭의 역할을 정의하고, "바로 발주" UI 위치를 결정하는 UX 의사결정 문서
>
> **Project**: DenJOY (치과 임플란트 재고관리 SaaS)
> **Date**: 2026-03-03
> **Status**: Draft — 의사결정 대기

---

## 1. 현황 분석

### 1.1 현재 모바일 앱 구조

**하단 탭 네비게이션** (5개 탭):
- 홈 (`overview`)
- **재고** (`inventory_master`) ← 검토 대상
- **주문** (`order_management`) ← 검토 대상
- 교환 (`fail_management`)
- 더보기 (메뉴)

**파일 위치**:
- `components/dashboard/MobileDashboardNav.tsx` — 모바일 탭 렌더링
- `App.tsx` 라인 298 — `mobilePrimaryTabs` 정의
- 타입: `/types.ts` — `DashboardTab`, `InventoryItem`, `Order` 정의

### 1.2 현재 "재고" 탭 (inventory_master) 동작

**상단 섹션**:
- 발주 필요 요약 (N종 · 총 M개 부족)

**하단 섹션** ("주문 필요 목록"):
- 품목 리스트 (제조사, 브랜드, 규격, 부족량)
- **각 품목마다 "바로 발주" 버튼 포함** ← 주문 탭으로 가지 않고 즉시 발주 가능

### 1.3 현재 "주문" 탭 (order_management) 동작

- 발주 내역 관리 (생성, 취소, 받음 처리)
- 상태별 필터, 삭제 등
- 이력 조회

---

## 2. 사용자가 제안한 방향

**"재고 탭에서 발주 UI 제거 → 역할 일원화"**

- 재고 탭: **재고 현황 + 최적화 정보만** 보여주기
- 주문 탭: **발주 관련 모든 작업** 일원화 (생성 포함)
- 장점: 화면 깔끔함, 기능 응집성 향상

---

## 3. 의사결정 검토항목

### 3.1 역할 분리 관점 (UX 아키텍처)

**제안된 모델 (Tab-Focused)**:
| 탭 | 역할 | 콘텐츠 |
|-----|------|--------|
| 재고 | 현황/분석 | 현재 재고량, 경고, 추천재고, 품목 최적화 |
| 주문 | 발주 관리 | 부족 목록 + 바로 발주, 발주 이력, 취소 |

**장점**:
- 기능 응집도 높음 (정보 보기 vs. 액션)
- 웹 대시보드와 구조 일관성 (각 탭이 단일 책임)
- 사용자가 "내가 무엇을 하려는가" → 탭 선택 명확

**단점**:
- 재고 확인 후 발주까지 탭 변경 필요
- 신입 운영자: 부족한 품목을 어디서 발주하는지 헷갈릴 수 있음

---

### 3.2 워크플로우 관점 (병원 운영자 관점)

**실제 업무 시나리오**:

1. **일과 시작: 재고 상태 확인**
   - "지금 재고 상황이 어떻게 되나?" → 재고 탭
   - 현황 (수량, 경고, 추세) 확인

2. **발주 필요 시: 주문 생성**
   - "부족한 것들을 발주하자" → 주문 탭
   - 부족 목록 보기 + 체크박스로 선택 + 한번에 발주

**평가**:
- 현재 ("재고" 탭에서 바로 발주): **1회 탭**
- 제안된 구조 ("주문" 탭으로 이동): **2회 탭**
- 비용: 미미함. 그러나 **심리적 일관성 향상** 가능

**병원 운영자의 평가** (요청자):
> "모바일 재고 탭에서는 발주 UI 없애고... 발주는 주문 탭 하나에서만 관리하도록 일원화하자."

→ 이미 이 방향을 **선호**하는 상태

---

### 3.3 "재고" 탭에 무엇을 보여줄 것인가?

**현재** (발주 버튼 포함):
- 요약 (N종 · M개 부족)
- 부족 목록 (발주 버튼)

**제안 후** (발주 버튼 제거):
| 우선순위 | 항목 | 용도 | 필수 |
|----------|------|------|------|
| P0 | 현재 재고량 (품목별) | 각 제품의 정확한 수량 확인 | ✅ Yes |
| P0 | 경고/부족 마크 | 부족 상황 시각화 | ✅ Yes |
| P1 | 기초재고(안전재고) 설정 | 경고 기준선 확인 | Should |
| P1 | 월 평균 소진량 | 추세 파악 | Should |
| P1 | 추천발주량 계산 | 품목별 ROP 제시 | Should |
| P2 | 조정 이력 | 언제 얼마나 조정했나? | Could |

**추천 MVP** (발주 버튼 제거 후):
```
상단: 요약 (부족 N종, 총 M개)
하단: 품목 테이블
  - 제조사, 브랜드, 규격
  - 현재 재고
  - 기초재고 대비 상태 (경고 배지)
  - [필요시] 월 소진량
```

---

### 3.4 "바로 발주" 편의성 유지 방안

**절충안**:

#### 옵션 A: 현재 상태 유지 (권장하지 않음)
- 재고 탭에 "바로 발주" 버튼 계속 배치
- 문제: 운영자가 제안했으므로 불채택 위험

#### 옵션 B: 제안 수용 (강한 일관성)
- 재고 탭 → 발주 정보만 보기
- 주문 탭 → 부족 목록 + 발주 기능 일원화
- **모바일 → 주문 탭 진입 시간: 1회 탭**
- 편의성 손실: 미미 (탭 이동 1회)

#### 옵션 C: 스마트 버튼 (절충)
- 재고 탭에 **"부족한 품목을 주문 탭에서 발주하기"** 버튼 (탭 이동 유도)
- 장점: 발주 프로세스를 명시적으로 안내
- 단점: 추가 UI 요소, 여전히 탭 이동 필요

#### 옵션 D: 클릭 후 모달 (사용성 개선)
- 재고 탭에 "부족한 항목" 항목 클릭 → 발주 모달 팝업
  - 모달 내: 품목 확인 + 수량 입력 + 발주 버튼
  - 완료 후 모달 닫기 → 재고 탭으로 복귀
- 장점: **탭 이동 없음**, 개별 발주 가능
- 단점: 여러 품목 한번에 발주 어려움, 발주 이력 보려면 탭 이동 필요

## 4. 권장 방향 (의사결정)

### 4.1 최종 권장안: **옵션 B (제안 수용)**

**선택 이유**:

1. **운영자의 명확한 의견** — 역할 분리 일원화 원함
2. **기능 응집도** — 재고 보기 vs. 발주 관리 명확 분리
3. **웹 대시보드와의 일관성** — 웹도 탭별로 기능 분리
4. **스케일 가능성** — 향후 기능 추가 시 각 탭에서 관련 기능만 증가
5. **학습곡선** — 신입도 직관적 (재고보기 → 주문탭에서 발주)

**구체적 스펙**:

#### "재고" 탭 (inventory_master)
```
┌─────────────────────────────────┐
│ 요약 정보 (상단)                  │
│  "부족 3종 · 총 12개 필요"       │
│                                  │
├─────────────────────────────────┤
│ 재고 현황 테이블 (하단)           │
│  제조사 | 브랜드 | 규격 | 현수량 │
│  기초재고 | 상태(⚠️)              │
└─────────────────────────────────┘
```

#### "주문" 탭 (order_management)
```
┌─────────────────────────────────┐
│ 발주 필요 목록 (상단)             │
│  □ 오스템 6/5.8 (부족 5개)       │
│  □ 덴티움 5/7 (부족 3개)         │
│  □ 메가젠 4.5/11 (부족 4개)      │
│ [발주하기] 버튼                   │
│                                  │
├─────────────────────────────────┤
│ 발주 이력 (하단)                  │
│  [2026-03-02] 발주 완료 ...      │
│  [2026-02-28] 받음 ...           │
└─────────────────────────────────┘
```

#### 사용자 워크플로우

```
1. 모바일 앱 진입
   ↓
2. "재고" 탭 탭 → 현황 확인 (수량, 경고)
   ↓
3. 발주 필요 → "주문" 탭 이동
   ↓
4. "부족한 항목" 체크 + "발주하기" 버튼
   ↓
5. 발주 완료 → "발주 이력"에 표시
```

## 5. 구현 범위

### In Scope (Must)
- [ ] 재고 탭에서 "바로 발주" 버튼 제거
- [ ] 주문 탭에 "부족한 품목 목록" 섹션 추가
  - 초기값: 현재 부족한 품목 자동 체크
  - 체크/언체크로 발주 품목 선택
- [ ] 모바일 레이아웃 최적화 (각 탭의 카드/테이블 반응형)
- [ ] 재고 탭 타이틀 변경 (또는 그대로 유지)

### Should Have
- [ ] 주문 탭 진입 시 부족 목록 자동 스크롤 (추천 영역)
- [ ] 완료된 발주의 경우 "부족한 품목"에서 자동 제외
- [ ] 재고 탭에 미니 추천재고 표시 (기초재고 대비 %)

### Could Have
- [ ] 재고 탭: 월 평균 소진량 표시
- [ ] 주문 탭: "한번에 모두 발주" 빠른 버튼
- [ ] 품목별 발주 설정 (선호 수량, 최적발주량 저장)

### Won't Have
- [ ] 자동 발주 (ML 기반)
- [ ] 재고 탭에 수정 기능 (수정은 "더보기" → 기초재고 설정)

| 항목 | 기준 | 측정 방법 |
|------|------|----------|
| 기능 완성도 | 재고/주문 탭 분리 완료, 모바일 발주 가능 | 기능 체크리스트 |
| 사용성 개선 | 기존 대비 UI 복잡도 감소, 탭별 기능 명확 | 스크린샷 비교, 운영자 피드백 |
| 웹-모바일 일관성 | 모바일 탭 구조 = 웹 대시보드 탭 역할 | 문서 검증 |
| 성능 | 탭 전환 시간 < 300ms, 부족 목록 렌더링 < 500ms | 성능 측정 |

## 7. 리스크 및 완화

| 리스크 | 영향 | 확률 | 완화 방안 |
|--------|------|------|---------|
| 기존 사용자 적응 | 발주 버튼 위치 변경 → 혼란 가능 | 중 | 출시 노트에서 명확히 안내, 상담 준비 |
| 부족 목록 업데이트 지연 | 재고 탭에서 확인 후 주문 탭 진입 시 수량 변경 → 부족 데이터 불일치 | 저 | Realtime subscription으로 실시간 동기화 |
| 모바일 레이아웃 공간 부족 | 주문 탭에 부족 목록 + 이력 → 스크롤 너무 길어짐 | 중 | 탭 분리 (부족 목록 따로 부분 탭) 또는 아코디언 접기 |

## 결론 (300자 이내)

**현재 제안 수용 + 단계적 개선 권장**.

재고 탭에서 발주 UI 제거하고, 주문 탭을 발주 허브로 일원화하면 기능 응집도 향상되고 웹 대시보드와의 일관성도 높아집니다. 탭 이동 1회 추가되지만, 사용자 경험상 충분히 명확합니다. 다만 주문 탭이 길어질 수 있으므로, 초기 발주-이력 섹션 분리 또는 아코디언 패턴 검토 필요.
